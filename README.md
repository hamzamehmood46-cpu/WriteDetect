# ✍️ WriteDetect — AI Origin Detector

**Project 2 — Integrating Large Language Models and Deploying AI Applications**
Introduction to AI Agents, Summer I 2026

## Problem

Text generated by large language models is now nearly indistinguishable from human writing at a glance, which makes plagiarism detection, academic integrity, and content moderation harder than ever. WriteDetect tackles this directly: given any piece of text, it predicts whether a human or an AI system wrote it, and explains *why* — combining six trained classifiers with two Hugging Face LLMs that add a second opinion and a plain-English explanation.

This builds on the Project 1 submission (classical ML + deep learning detectors) and extends it with LLM-based reasoning, per the Project 2 brief.

## Dataset

| Property | Value |
|---|---|
| File | `train_data with labels.xlsx` |
| Columns | `text`, `label` |
| Total samples | 8,176 |
| Human Written (label 0) | 4,088 |
| AI Generated (label 1) | 4,088 |
| AI sources | GPT, Claude, Mistral, LLaMA, Falcon |
| Split | Stratified 80/10/10 (ML); 5-fold CV (DL) |

The dataset is perfectly balanced between human-authored and AI-generated text, sourced from a mix of major LLM providers to reduce single-model bias. A small reference copy (`data/AI_vs_huam_train_dataset.xlsx`) and sample test essays (`data/test_data/`) are included in this repo.

### Extended dataset (AI Judge fine-tuning)

The LoRA fine-tuning of the AI Judge LLM (see `finetune_ai_judge_colab.ipynb`) uses a much larger public dataset instead: the Kaggle ["AI vs Human Text"](https://www.kaggle.com/datasets/shanegerami/ai-vs-human-text) dataset (`AI_Human.csv`, 487,235 rows). It isn't included in this repo due to its size (350MB) — download it from Kaggle directly if you want to re-run the fine-tuning notebook.

## Models Used

### Classical machine learning (Project 1, retained)
- **Support Vector Machine** — linear kernel, TF-IDF features (unigrams + bigrams)
- **Decision Tree** — TF-IDF features, max depth 20
- **AdaBoost** — 100 estimators over TF-IDF features

### Deep learning (Project 1, retained)
- **Bidirectional GRU** — Word2Vec-100d embeddings trained from scratch
- **LSTM** — GloVe-100d pre-trained embeddings
- **TextCNN** — multi-filter convolutions over Word2Vec-100d embeddings

### Large language models (new in Project 2)
| Model | Hugging Face ID | Role |
|---|---|---|
| AI Judge | `Qwen/Qwen2.5-0.5B-Instruct` | Independently classifies the text as Human-Written or AI-Generated and gives a short reasoning — a genuine second opinion, not derived from the ML/DL models' output. |
| Plain-English Explainer | `HuggingFaceTB/SmolLM2-360M-Instruct` | Takes the classifier's verdict, confidence score, and top LIME keywords, and rewrites them as a short, jargon-free explanation for non-technical users. |

Both LLMs are loaded through Hugging Face `transformers` chat-style pipelines and were chosen specifically for their small size (0.5B and 360M parameters) from two different model families, so the app stays responsive on the free CPU tier of Hugging Face Spaces.

### Fine-tuning the AI Judge (extra credit)

The app supports swapping the base AI Judge for a LoRA fine-tuned version, trained on WriteDetect's own labeled dataset instead of relying on zero-shot prompting alone. A dropdown on the **Analyze Text** and **Upload Document** pages lets you pick "Base (zero-shot)" or "Fine-tuned (LoRA on WriteDetect data)" — the fine-tuned option only appears once an adapter has been trained.

To train it:

1. Run `venv\Scripts\python prepare_finetune_data.py` locally — this exports a balanced, chat-formatted JSONL dataset (`finetune_data/ai_judge_train.jsonl` and `ai_judge_val.jsonl`) from the labeled training data.
2. Upload `finetune_ai_judge.py` and both JSONL files to **Google Colab** (free T4 GPU; CPU training is impractically slow for this) or **TTU HPCC**.
3. On Colab, install dependencies and run the script:
   ```
   !pip install -q transformers peft accelerate datasets
   !python finetune_ai_judge.py
   ```
4. The script LoRA fine-tunes `Qwen/Qwen2.5-0.5B-Instruct` (only ~1M trainable parameters via LoRA, r=8) to predict the Verdict/Reasoning format directly from the training labels, then saves and zips the adapter as `qwen_judge_lora.zip`.
5. Download the zip, unzip it, and copy its contents into `models/qwen_judge_lora/` in this project. Restart the app — the "Fine-tuned" option will now appear in the dropdown.

The adapter is tiny (a few MB, LoRA-only) since the base model weights aren't duplicated — safe to commit to the repo or upload alongside the other model files when deploying to Hugging Face Spaces.

## Results

| Model | Notes |
|---|---|
| SVM | Highest accuracy among classical models; linear decision boundary works well on TF-IDF features |
| Decision Tree | Most interpretable but lower accuracy and prone to overfitting at depth 20 |
| AdaBoost | Improves over a single Decision Tree by boosting weak learners |
| Bidirectional GRU | Captures sequential context better than the classical models |
| LSTM | Comparable to the GRU, benefits from pre-trained GloVe vectors |
| TextCNN | Fastest deep learning model, competitive accuracy via local n-gram filters |
| Qwen2.5 AI Judge | Agrees with the trained classifiers on most clearly AI-generated text; more useful as a sanity check on ambiguous or short inputs |
| SmolLM2 Explainer | Reliably turns numeric confidence + keywords into a readable sentence for non-technical users |

ROC curves, confusion matrices, and a training word cloud are available in the `plots/` folder and rendered on the **About Models** page of the app.

## What We Learned

- TF-IDF + classical ML remains a strong, lightweight baseline for authorship detection and is much cheaper to run than the deep learning models.
- Word embeddings (Word2Vec, GloVe) give the recurrent and convolutional models a meaningful boost in capturing stylistic patterns beyond bag-of-words.
- Small instruction-tuned LLMs (sub-1B parameters) are enough to produce a coherent independent judgment and a readable explanation — a full-size LLM isn't necessary for this kind of auxiliary reasoning task.
- Deploying multiple heavy frameworks together (PyTorch, TensorFlow, Transformers) on a free-tier CPU Space requires careful dependency pinning and lazy model loading (`st.cache_resource`) to keep startup time and memory usage manageable.
- LIME explanations and LLM-generated explanations serve different audiences: LIME is useful for technical users inspecting feature weights, while the LLM explanation is far more accessible to a general audience.
- LoRA makes fine-tuning a 0.5B model tractable on a single free-tier Colab GPU in minutes, training under 1M parameters instead of the full ~500M — but it genuinely needs a GPU; the same job is impractically slow on CPU.

## Setup

Python 3.11 is required (PyTorch/TensorFlow are not compatible with Python 3.12+).

```bash
py -3.11 -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
```

If `models/` is empty, train the ML models with:

```bash
venv\Scripts\python retrain_ml_models.py
```

Deep learning models (GRU, LSTM, TextCNN) require running the full training notebook. The LLMs (`Qwen2.5-0.5B-Instruct`, `SmolLM2-360M-Instruct`) download automatically from Hugging Face on first use — no separate training step is required.

## Run Locally

```bash
venv\Scripts\streamlit run app.py
```

Open `http://localhost:8501`.

## Deploying to Hugging Face Spaces

1. Create a new Space at [huggingface.co/new-space](https://huggingface.co/new-space), choosing **Streamlit** as the SDK.
2. Clone the Space repo and copy in `app.py`, `llm_utils.py`, `models_dl.py`, `utils.py`, `requirements.txt`, `models/`, and `plots/`.
3. Commit and push — the front matter at the top of this README configures the Space automatically.
4. Wait for the build to finish, then open the public Space URL.

The two LLMs download automatically from the Hugging Face Hub at runtime, so no extra upload is needed for them — only the trained ML/DL model files in `models/` need to be pushed (use `git lfs track` for files over ~10MB).
